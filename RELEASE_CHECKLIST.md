# 🚀 Focus Flow 출시 체크리스트 (Release Checklist)

출시 전, 아래 항목들을 반드시 순차적으로 확인하고 업데이트합니다.

## 0. ⚠️ 에이전트 필수 준수 규칙 (절대 엄수)
- [ ] **버전 임의 상향 금지**: `versionCode` 및 `versionName`은 개발자의 명시적 요청 없이 절대 수정하지 않는다.
- [ ] **배포 승인 필수**: 모든 Git 작업(Push, Tag 등)은 반드시 개발자에게 보고하고 최종 승인을 받은 상태에서만 수행한다.
- [ ] **체크리스트 수정 금지**: 릴리즈 체크리스트는 개발자의 명시적 동의 없이 절대 수정하거나 내용을 변경하지 않는다.

## 1. 버전 및 정보 업데이트
- [ ] **build.gradle.kts**: `versionCode` 상향 및 `versionName` 최신화.
- [ ] **README.md**: 최신 버전 정보 업데이트 및 **이번 버전의 새로운 기능(NEW/FIX)**을 작성. 
- [ ] **릴리즈 기록의 독립성**: README.md의 '주요 기능' 섹션은 매번 덮어씌워지므로, **과거의 릴리즈 기록이 소실되지 않도록 별도의 히스토리 문서나 태그 메시지에 이번 버전의 구체적인 변경 사항(What & Why)을 기재**했는지 확인. 단순히 "README 참조"로 갈음하는 것을 금지함.
- [ ] **UI 표시 확인**: 스플래시 화면, 설정 탭 등 UI 내 버전 정보가 최신인지 확인.

## 2. 릴리즈 노트 작성 (⚠️ 중요: 이전 내용 복사 금지)
- [ ] **변경 로그 분석**: 지난 태그 이후의 Git 로그를 기반으로 실제 추가/수정된 사항 정리.
- [ ] **배포용 문구 작성**: 사용자가 읽기 쉬운 이모지와 함께 **이번 릴리즈의 핵심 변경점**을 5~7개 내외로 요약.
- [ ] **설치 안내**: APK 다운로드 및 업데이트 방법 포함.

## 3. 코드 및 안정성 확인
- [ ] **디폴트 값 확인**: `SettingsRepository` 등의 기본값이 기획 의도와 일치하는지 확인.
- [ ] **영속성**: 앱 업데이트 시 설정 데이터 유지 로직 검증.
- [ ] **시스템 채널**: 알림 채널 설정이 의도대로 시스템 소리를 차단하는지 확인.

## 4. Git 및 배포
- [ ] **Git Push**: 모든 변경 사항 푸시.
- [ ] **Git Tag**: 버전 네임 태그 생성 및 푸시 (`git tag v1.x.x`).
- [ ] **Github Release**: 태그(`v1.x.x`) 푸시 시 **GitHub Actions가 자동으로 실행**되어 `README.md`의 내용을 기반으로 릴리즈 노트를 작성하고 APK를 업로드합니다.

---
*이 체크리스트는 모든 릴리즈 작업의 기본 지침이며, 매 버전마다 "신선한" 정보를 제공해야 합니다.*
